# keel v1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship a working, tested core of keel - a durable execution runtime for TypeScript AI agents that replays completed steps after a crash, retries failures, and records token and cost per step.

**Architecture:** A core engine invokes a user-defined async workflow handler with a context object. Each `ctx.step` consults a pluggable store before executing; completed steps replay from the store instead of re-running. Stores and LLM providers sit behind interfaces so zero-cost local defaults (file store, Ollama) ship now and Redis/SQLite/OpenAI adapters drop in later.

**Tech Stack:** TypeScript, Node 20+, Vitest, zero runtime dependencies in v1 core (file store is JSON; Ollama is plain fetch).

## Global Constraints

- Zero money: no paid services, no paid LLM keys. Default provider is Ollama (local, free). Tests use a mock provider.
- No login/hosting: runtime and future dashboard are local-only. Nothing deployed.
- Zero runtime dependencies in the v1 core. Dev dependencies only (typescript, vitest, @types/node).
- Node 20 and 22 supported. ESM modules.
- Style: plain ASCII only in code, comments, docs. No em-dashes, no emoji, no smart quotes.
- Determinism contract: nondeterminism (network, time, random, model calls) must live inside `ctx.step`.

### Task 2: Store interface and MemoryStore

**Files:** Create `src/store/types.ts`, `src/store/memory.ts`, `src/store/index.ts`, `tests/store.test.ts`.

**Interfaces produced:**
- `RunRecord`, `StepRecord` types (fields per spec).
- `Store` interface: `createRun`, `getRun`, `updateRun`, `getStep`, `saveStep`, `listRuns`, `listSteps`.
- `MemoryStore implements Store`.

**Deliverable:** MemoryStore round-trips runs and steps. Test: create a run, save a step, read it back, list returns it.

### Task 3: FileStore (durable across restarts)

**Files:** Create `src/store/file.ts`; extend `tests/store.test.ts`.

**Interfaces produced:** `FileStore implements Store` backed by a JSON file, atomic write (temp file then rename).

**Deliverable:** Test simulates a restart: write with one FileStore instance, construct a fresh instance on the same path, read the data back. This is the crash-durability primitive.

### Task 4: Retry policy

**Files:** Create `src/retry.ts`, `tests/retry.test.ts`.

**Interfaces produced:** `RetryPolicy` (maxAttempts, baseMs, factor, jitter, retryable). `runWithRetry(fn, policy, onAttempt)` returning the result or throwing after maxAttempts.

**Deliverable:** Test - a fn that throws twice then returns succeeds at attempt 3; a fn that always throws fails after maxAttempts; attempt callback fires per try. Use injected sleep to avoid real delays.

### Task 5: Workflow definition and context

**Files:** Create `src/workflow.ts`, `src/context.ts`, `src/errors.ts`.

**Interfaces produced:**
- `defineWorkflow(name, handler)` returning a `WorkflowDefinition`.
- `WorkflowContext` with `step(name, fn, opts?)`, `sleep(name, ms)`, and `llm(name, args)`.
- `PausedError`, `StepFailedError`.

**Deliverable:** Types compile; context methods delegate to the runtime (wired in Task 6).

### Task 6: Runtime - run, replay, resume (the core)

**Files:** Create `src/runtime.ts`; `tests/memoization.test.ts`, `tests/recovery.test.ts`.

**Interfaces produced:** `Keel` class constructed with `{ store, provider? }`. `run(def, input)` and `resume(runId)` returning the run result.

**Core logic:** maintain per-run step cursor. On `ctx.step(name, fn)`: read StepRecord; if completed, return its result (replay); else run via retry, persist, return. On `ctx.sleep`: persist wake-at; replay returns immediately if elapsed. On handler throw: mark run failed (or paused for PausedError).

**Deliverables (tests):**
- Memoization: step with a side-effect counter runs once; second `run`/replay on the same store and runId does not re-run it.
- Recovery: workflow throws after step 1; `resume` does not re-run step 1; step 2 completes; status completed.

### Task 7: Provider interface, MockProvider, OllamaProvider, ctx.llm

**Files:** Create `src/providers/types.ts`, `src/providers/mock.ts`, `src/providers/ollama.ts`, `src/providers/index.ts`; `tests/provider.test.ts`.

**Interfaces produced:** `Provider.complete({model, prompt})` returning `{text, tokensIn, tokensOut, costUsd?}`. `ctx.llm(name, {prompt, model})` wraps a step, calls the provider, records tokens on the StepRecord.

**Deliverables (tests):** MockProvider drives ctx.llm; assert text + token capture; assert the llm step is memoized on replay (provider called once across two runs). OllamaProvider posts to localhost:11434 (not hit in CI; covered by a shape/unit test of request building).
